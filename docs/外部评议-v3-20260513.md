## 1. 项目一句话

我在做一个**多站群多 agent 平台**,内部跑 N 个独立站(每站由不同投手负责),平台是发动机,站是车壳。半个月到一个月会开始卖站给客户,卖源码 + 客户能持续从主平台拉升级。

**业务背景:**
- 单一品类:成人付费,折扣站定位(价格驱动 + 时效驱动)
- 流量打法:Meta/Google paid-ads 为主
- 投手在自己的站上工作,不互相干扰,不写代码,不写 agent
- agent 由平台统一升级,所有站同步生效

## 2. 当前已定的核心架构(只给你做上下文,不需要质疑)

### 2.1 形态
- **monorepo:** `multisite-platform/` 一个仓库装下 `platform-core/`(发动机)+ `sites/{site_id}/`(每站一个目录)+ `tools/customer-export/`(卖站剥离脚本,P1 工程项)
- **agent 按角色分层:** 通用(分析/策略/告警) + 投放 + 承接 + 技术,共 6 个 agent
- **能力包双层:**
  - 打法包(`packages/打法包-投手能力/`)= 投手各一份,决定"装哪几个 agent + 用哪套参数 + 走哪个 flow",可 fork,带 lineage
  - 产出包(`packages/产出包-自动化生产/`)= 跨投手共用的自动化生产流水线,5 个子包(脚本/分镜/视觉/旁白/合成),可单跑 / 链式跑 / 人工插桩
  - **新增第三层 内容包(刚定,还没设计)**:投手的内容调性偏好(限时紧迫 / 划算等品牌话术),独立维护,跨站复用

### 2.2 接口与合约
- 单 agent 调用入参:`site_id + time_window + upstream_output + overrides`
- 出参:`status(success/no_data/external_failure/timeout/failed) + data + cost + gap_reason`
- 参数覆盖优先级 4 层:**L1 平台默认 → L2 打法包 → L3 站级 → L4 运行时临时**
- 站级配置加载点:`sites/{site_id}/config.yaml`,runtime 启动 + 每次 agent 调用前扫描

### 2.3 数据层
- 每个子站有 `data-inbox/`(只丢不处理)
- 总站有 `ingestion-service/`(定时扫描 + 写入)
- 物理存储 MVP 阶段单库,site_id 逻辑隔离
- SLA:DB 99.9% / 数据延迟 ≤ 2 分钟

### 2.4 卖站给客户(关键)
- 卖**代码**:剥 `sites/{客户站}/` + `platform-core/` 子集打包成独立 repo,客户拿走
- 不剥**数据**:客户数据**回传到平台主数据库**(SaaS-like 模式)
- 客户走 sync 脚本从平台主仓库拉 `platform-core/` 升级,他改的 `site/` 不被覆盖
- platform-core 必须自洽(不能 import sites/),sites 调 platform-core 走稳定接口
- 客户和内部站共用一套 platform-core 代码 + 一套 data-spine

### 2.5 财务与预算
- 财务发**平台总监控台账号**(只读成本面板)
- 开站时财务设预算红线写进站 config(daily_cap / monthly_cap)
- 超限自动停 + 推财务,审批走平台内工作流
- 产出包月度成本独立闸门(单站 $50 / 月默认)

## 4. 当前悬空的两个核心问题(主要请你给建议)

### 🔵 问题 A · 控制台的形态选什么

**为什么必须现在拍:**
所有"写"动作(投手挂打法包 / 财务改预算 / 二次审批 / 提额审批)都要走控制台。控制台不定,后续 §6 监控台细化、卖客户的鉴权设计、控制台是否独立 repo 都没法继续推。

**MVP 阶段规模:** 2-3 个站,2-3 个投手,1 个财务,1 个 ZJB(老板)。
**目标规模:** 半年内 10 个站,5 个投手,3-5 个客户。

**三个候选,各有代价:**

#### 选项 A · Web 全栈控制台
- 形态:`platform-core/console-backend/` (Python API) + `web/` (React/Vue 前端)
- 优点:用户体验最好;能做权限分层、审批流、可视化;以后卖客户能直接给一个"客户控制台"
- 代价:工程量最大,前后端都要建栈,投手 0 工程能力前先要等控制台出来才能用
- MVP 时间估计:2-4 周建出最小可用版

#### 选项 B · API 先行,UI 后补
- 形态:只做 `platform-core/api/`,投手通过 Postman / curl / 简单脚本调
- 优点:工程量小,先把"写入口的 schema"定下来,UI 后期补
- 代价:**投手 0 工程能力,不可能用 curl** —— 实际等于阻塞投手作业
- 不可行,除非 MVP 阶段所有写入操作都由 ZJB 代投手做

#### 选项 C · 低代码(Streamlit / Retool)
- 形态:用 Streamlit / Retool 之类快速搭一个控制台
- 优点:1-3 天能跑出能用的控制台
- 代价:卖客户时这种工具不能直接给客户用(授权 / 品牌 / 集成);后期还是要重写成 A
- 适合:MVP 内部用,但要有"以后扔掉重写"的觉悟

#### 我的倾向
**A**,理由:控制台是这个平台的真实门面,投手 / 财务 / 客户**都通过它和平台交互**,工程量虽大但绕不开。低代码省时但卖客户重写一次成本更高。

**但我不确定 MVP 阶段直接上 A 是不是过度设计,以及前端栈选什么(React + Next.js? 还是更轻的 Vue/Astro?)。**

---

### 🔵 问题 B · 客户数据回传后怎么隔离

**背景:** 客户拿源码独立部署,但**新产生的数据回传到平台主数据库**。这意味着平台主库里同时有**内部站数据 + N 个客户站数据**。

**当前 MVP 已采用:** 单库 + 单 schema + `site_id` 字段做逻辑隔离。

**问题:** 卖客户后,逻辑隔离够不够?

#### 候选 1 · 维持现状(逻辑隔离)
- 同库 / 同 schema / `site_id` 区分
- 优点:简单,所有 agent 代码不改;跨站统计天然支持
- 风险:
  - 一个 SQL bug 漏 `WHERE site_id = ?` 就泄客户数据 → 客户 A 看到客户 B 的数据
  - 客户合规审计可能要求物理隔离(GDPR / SOC2 等)
  - 备份恢复粒度粗,某一个客户要清数据要扫全表

#### 候选 2 · Schema 隔离
- 同库,但每客户独立 schema(`customer_a.sessions / customer_b.sessions`)
- 内部站走 `internal.sessions`
- 优点:Postgres schema 切换轻量,SQL 安全性大幅提升
- 代价:跨客户统计要 UNION,ORM 配置复杂

#### 候选 3 · 物理隔离(独立 DB 实例)
- 每客户独立 Postgres 实例
- 优点:合规友好,客户事故不影响其他客户
- 代价:运维成本爆炸,跨客户分析做不了
- 一般是企业级 SaaS(Salesforce / Snowflake)的玩法

#### 候选 4 · 读时隔离 + 写时合一(混合)
- 物理同库,但应用层强制每个 query 必须带 `site_id`,通过 ORM / 查询封装强制
- 备份按 site_id 拆分快照
- 安全权限细到字段级
- 优点:享受逻辑隔离的灵活,补强了候选 1 的安全短板
- 代价:对 ORM / 查询层有改造要求

#### 我的倾向
- MVP 阶段(还没卖客户): 候选 1
- 第一个客户上线前:升到候选 2 或候选 4
- 候选 3 太重,不是这个阶段需要的

**但我不确定:** "读时隔离 + 写时合一"在 Python + Postgres 生态下的成熟方案是什么?有没有踩过坑的工程案例?
