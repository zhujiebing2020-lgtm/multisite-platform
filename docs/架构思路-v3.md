## 0. 一句话定位

**平台是"发动机",站是"车壳",能力包是"投手的打法 + 生产流水线"。**
- 发动机 = 6 个 agent(照抄 crave-ai 现有角色切分)
- 打法包 = 投手各自的 agent 组合 + 参数偏好
- 产出包 = 跨投手共用的自动化生产流水线(对品类无感)
- crave-ai 本身不动,平台从它**读逻辑 → 在新 repo 重写**

## 3. 能力包层(v3 新增 · 核心)

### 3.1 两类能力包并列

```
platform-core-平台内核/
  └─ packages-能力包/
      │
      ├─ 打法包-投手能力/                决策"怎么打" · 每投手一份
      │   ├─ 投手A-激进付费.yaml
      │   ├─ 投手B-稳健承接.yaml
      │   └─ ...
      │
      └─ 产出包-自动化生产/              执行"怎么造" · 跨投手共用 · 对品类无感
          ├─ ad-video-广告视频/
          │   ├─ script-脚本.yaml
          │   ├─ storyboard-分镜.yaml   ← 平台提供"分镜"能力位,内容由投手填
          │   ├─ visual-视觉.yaml
          │   ├─ voiceover-旁白.yaml
          │   └─ synthesis-合成.yaml
          ├─ landing-落地页/(未来)
          └─ comment-评论区/(未来)
```

### 3.2 打法包 = agent 组合方案

每个投手一份,决定"装哪几个 agent · 用哪套参数 · 走哪个 flow"。

```yaml
# packages-能力包/打法包-投手能力/投手A-成人付费-激进.yaml
_meta:
  author: 投手A
  category: 成人付费                 # 强制 · 挂载时校验站 category 匹配
  stage: 激进                         # 起步/稳健/激进/收割
  created: 2026-05-10
  forked_from: 投手B-成人付费-稳健@v2.3    # fork 链,没 fork 就留空
  lineage:
    - 投手B-成人付费-稳健@v2.3
    - 投手A-成人付费-激进@v2.4
    - 投手A-成人付费-激进@v2.5(当前)
  version: 2.5

enabled_roles: [通用-common, 投放-ads]
active_flow: paid-ads-付费流量

thresholds_overrides:    # 投手的参数偏好,覆盖平台默认
  cphq_green_max: 12
  learning_to_run_spend: 50
  pause_on_hvu_zero_days: 1   # 比默认更激进
```

**命名规范强制:** `投手×品类×风格.yaml`,例 `投手A-成人付费-激进.yaml`。
缺任一维度 → 系统拒绝加载。防止投手 A 的成人付费激进打法被误挂到带货新站。

### 3.3 产出包 = 自动化生产流水线

跨投手共用,跨品类通用 —— **对品类无感**。

```yaml
# packages-能力包/产出包-自动化生产/ad-video-广告视频/storyboard-分镜.yaml
_meta:
  category_agnostic: true     # 明写:对品类无感
  human_gate: optional         # 可选人工审(投手决定是否必审)

inputs:
  - 素材库路径
  - 品类词(带货3C / 成人付费 / 教育 / ...)
  - 调性(专业 / 诙谐 / 温情)
  - 品牌规范

outputs:
  - storyboard.json            # 分镜结构化描述(结构由投手定义)

cost_tracking:
  model: claude-sonnet-4-6
  est_cost_usd: 0.3
```

**边界说明:** 产出包只提供"能力位",内容由投手填。
分镜里要不要锚、锚什么、怎么锚,是执行层决策,平台不预设。

子包可单独调或链式调:
- **单独调**:投手手动"给我生成一个分镜",只跑 storyboard
- **链式调**:策略 agent 判断"素材老化" → 触发整条 script→storyboard→visual→voiceover→synthesis
- **人工插桩**:任一子包后可挂 `human_gate: required`,必须人工审才能继续

### 3.4 站级配置同时挂两类

```yaml
# sites-站群/{site}/config.yaml
_meta:
  site_id: 站1
  owner: 投手A
  category: 成人付费

打法包: 投手A-成人付费-激进@v2.5
# 加载时系统校验:打法包._meta.category == 站._meta.category,不匹配拒绝挂载

产出包:
  - ad-video-广告视频          # 整条启用
  # 或细选:
  # - ad-video/storyboard-分镜(human_gate: required)

素材源: ./assets/

budget_limits:                 # 见 §4 财务
  daily_cap_usd: 500
  monthly_cap_usd: 12000
  breach_action: block
  breach_approver: 财务

creative_refresh:              # 见 §4.5 自动产出的成本闸门
  monthly_cap_usd: 50
  approval_window_h: 24
  on_timeout: 拒绝
```

### 3.5 边界对比(打法包 vs 产出包)

| | 打法包 | 产出包 |
|---|---|---|
| 做什么 | 决策:什么时候停 / 上 / 加 | 制造:原材料 → 成品 |
| 跨投手共用? | 不共用,每人一份 | 共用,所有投手调同一流水线 |
| 参数归属 | 投手判断偏好 | 素材库调性 + 站品类 |
| 站级可改吗 | 整包挂载,不改内部 | 可单独启用子包 |
| 对品类敏感? | 是(带货/成人差异大) | **否**(明写 category_agnostic) |

## 4. 财务角色与预算红线(v3 新增)

### 4.1 财务的边界

**不建新控制台,发一个平台总监控台账号。**

```
财务
  └─ 账号:平台总监控台(只读 · 成本口径面板)
  └─ 做:
      - 开站时设定预算红线 → 写进站 config.yaml
      - 成本异常告警复核
      - 月度成本账汇总
      - 审批投手的预算提额申请
  └─ 不做:
      - 不碰业务判断
      - 不改打法包
      - 不参与投放决策
```

### 4.2 预算红线流程(方案乙:先定再动)

```
【开站 Day 0】
  ZJB/运营 决定开站 + 分派投手
      ↓
  财务 设定预算红线 → 写进 sites-站群/{site}/config.yaml
      daily_cap_usd / monthly_cap_usd / breach_action: block
      ↓
  投手 接手配置打法包 / 产出包

【投手日常】
  红线内自由操作(不打扰财务)
  想突破 → 发起审批 → 财务点头 → 解锁临时 / 永久提额

【系统自动】
  实际花销 ≥ daily_cap → 投放 agent 停下 → 企微推给财务 + 投手
  财务决定:准 / 不准 → 平台记录留痕
```

### 4.3 审批走平台内工作流

```
投手点"申请提额" → 平台工单生成
                 → 企微/飞书推财务(带一键跳转)
                 → 财务在平台内点同意/驳回
                 → 回写到站 config.yaml,留审批日志
```

不用邮件、不用线下签字。工单、推送、决策、留痕**都在平台内闭环**。

### 4.4 成本账(产出包这边)

每次跑产出子包,成本写进 data-spine:

```
data-spine 成本事件:
  site_id · package · subpackage · model · tokens · cost_usd · timestamp
```

财务月度汇总:哪个站 / 哪个投手 / 哪类产出包 花了多少。

### 4.5 自动产出触发的成本闸门(v3 补丁 · 05-13)

产出包既可手动调也可被策略 agent 自动触发。自动触发必须带闸门,否则成本失控。

**触发口径具体化**(由 `workflows-工作流/creative-refresh-素材更新.yaml` 定义):
- 条件:CTR 连续衰减 / CPM 连续上升 / 转化衰减 等(具体阈值在工作流文件里)
- 冷却期:同一站 N 天内最多触发一次
- 审批窗口:投手必须在 X 小时内审,超时**默认拒绝**(不是默认通过)
- 月度产出预算:单站产出包花销有上限,超限暂停后续自动触发,推财务复核
- 二次审批:自动产出的成品进 `assets-pending/`,投手二次确认才进 `assets-ready/`

关键原则:
- **超时默认拒绝** —— 防止投手忘了审就把预算花光
- **预算闸门独立于 `budget_limits`** —— 投放预算和产出预算分开记账

## 5. 运营流程(v3 新增 · 验证架构对不对齐现实)

### 5.1 角色与职责

```
ZJB(老板/调度层)
  看:平台总监控台
  做:分配谁管哪站 · 打法包归档/审批 · 异常介入

运营(辅助调度)
  看:同 ZJB · 权限稍低
  做:日常派单 · 催进度 · 汇总周报

投手 A/B/C/D...(执行层 · 多人)
  看:名下站的监控台(自己的站完整权限)
  做:写打法包 · 挂到站上 · 调参数 · 提产出需求

素材组(独立)
  做:维护素材库 · 喂给产出包当原料

财务(§4)
  看:总监控台成本面板
  做:定预算红线 · 审提额 · 成本账

技术(ZJB 或外包)
  做:agent 原子层升级 · flow 调优 · 平台运维
```

### 5.2 一个新站从 0 到跑通

```
【Day 0】ZJB 决定开新站
  ├─ 定品类 / 域名 / 预算盘
  ├─ 分派给投手 B
  └─ 财务设预算红线

【Day 1~2】投手 B 接手
  ├─ 在 sites-站群/ 下建站目录(用模板)
  ├─ 挂打法包:投手B-稳健承接
  ├─ 挂产出包:ad-video(整条)+ landing(未来)
  ├─ 素材组接需求,放原料进素材库
  └─ 选 flow:paid-ads-付费流量

【Day 3】产出包跑第一批
  ├─ script → storyboard → 投手 B 人工审(锚不锚、怎么审,投手决策,平台不强制)
  ├─ 过了 → visual → voiceover → synthesis
  └─ 成品进 sites-站群/站X/assets-ready/

【Day 4】站上线
  ├─ 投放 agent 按打法包跑 Meta/Google
  ├─ 分析 agent 每天回数据
  ├─ 诊断 agent 异常推到工作台
  └─ 策略 agent 自动提第二批产出需求,投手 B 审批放量

【每日】
  ├─ 投手 B 开站群监控台,看名下 3~5 个站
  ├─ 告警 agent 半夜已经把异常推企微
  ├─ 投手 B 决策:停哪些组 / 加预算 / 要新素材
  └─ ZJB 只看总监控台红灯

【每周】
  ├─ 投手间打法交叉评审
  ├─ 打法包迭代(author 自己改 / 或 fork 出新版)
  └─ ZJB 拍板:哪些包进"官方推荐"池
```

### 5.3 特殊场景

| 场景 | 处理 |
|---|---|
| 投手 B 请假,B 的站谁管 | ZJB 临时把站重分派给投手 A,A 挂**自己的**打法包(不是用 B 的),站历史数据保留 |
| 投手离职 | 打法包还在 `packages/` 里,新人直接 fork 或拿来用,lineage 自动记录 |
| 带货站换成人付费站 | 不换产出包,只换输入源(品类词/调性/品牌规范)。流水线对品类无感 |
| 紧急预算突破 | 投手工单 → 平台推财务 → 财务决定 → 系统自动执行 |

# §11 data-spine 详细设计(v3 补丁 · 05-13 · P1-1 落地)

## 11.1 分层原则(抄 crave-ai 经验教训)

crave-ai 的血泪教训:**MCP/AdClaw 自动拉数会失败**,所以"投手手抓补数"是补丁。
data-spine 把这个分层设计成一等公民,不是补丁:

```
自动层 auto      三方 API 拉的数据(Meta/Google/AdClaw)       可能失败
补录层 manual    投手/吴玲/财务手抓的数据(JSON)              兜底
合并层 merged    两层合并后的标准视图,给 agent/监控台读       对上层透明
```

agent 只读 `merged`,不关心数据从哪来。

## 11.2 子站 / 总站分工(关键分层)

```
子站(sites-站群/站X/)
  职责:产出数据文件(JSON)
  路径:sites-站群/站X/data-inbox/{auto,manual}/
  不做:不监听、不处理、不写 data-spine
  投手做什么:把手抓数据丢进 data-inbox/manual/,丢完就走

总站(platform-core-平台内核/data-spine/ingestion-service/)
  职责:监听所有子站 data-inbox/,拉进 data-spine
  监听机制:定时扫描(乙方案)
  失败兜底:dead-letter-queue
```

**为什么这样分:**
- 子站轻,只产数据,格式错总站退回
- 总站重,脏活累活集中,升级只改总站
- 权限清晰,投手写不到 data-spine
- 故障隔离,总站挂了子站照攒,恢复后批量补
- 对应 crave-ai 现状(吴玲日报就是丢目录等脚本吃)

## 11.3 监听机制 · 定时扫描(方案乙)

**扫描策略三件套:**

1. **已处理归档** — 处理完的文件 move 到 `.processed/`,inbox 永远轻
2. **文件名幂等** — `yyyy-mm-dd-HHMMSS-{hash8}.json`,hash 见过就跳过
3. **自适应间隔** — 空闲 60 秒/次,有数据追平期 5 秒/次

**成本(假设 100 子站)**
- 单次扫描 ≈ 200ms
- 日常 CPU:5 分钟/天,<$0.05/月
- 1000 站以内不用换架构

**实现位置**
- 不开单独机器,当平台后台协程跑
- 和 agent runtime / 监控台共用资源

## 11.4 SLA(给别人用的底线)

| 项 | 目标 |
|---|---|
| DB 可用性 | 99.9%(月宕机 ≤ 43 分钟) |
| 数据延迟 | ≤ 2 分钟(监控台看到的不老于 2 分钟) |
| 外部依赖失败 | 标记为 null + gap_reason,绝不填 0 假数据 |
| 投手补录延迟 | ≤ 5 分钟可反映到监控台 |

## 11.5 异常处理分层

**平台内(零容忍)**
- 关键写入走事务 + retry,失败进 dead-letter-queue(不丢数据)
- agent 启动自检(DB 连通、磁盘空间、配置完整)
- 崩溃自动重启 + 恢复未完成调用

**平台外(Meta/Google/LLM · 标记清楚)**
- 超时 + 重试 3 次
- 3 次失败 → `agent_calls.status = 'external_failure'` + `error.reason`
- 指标 `value = NULL, gap_reason = 'meta_api_unavailable'`
- 告警推运营,不推投手(投手改不了外部问题)

# §12 agent 标准 I/O 格式(v3 补丁 · 05-13 · P1-2 落地)

## 12.1 入参(所有 agent 一致)

```yaml
input:
  site_id: "站1"                              # 谁的数据
  time_window:                                # 哪段时间
    from: "2026-05-01"
    to:   "2026-05-13"
  upstream_output: { ... }                    # 上游 agent 的 data 字段,没有就空
  overrides: { ... }                          # 可选配置覆盖,一般空
```

四个字段,agent A 和 agent B 入参长相完全一样。

## 12.2 出参(四字段,一个都不少)

```yaml
output:
  status: success                             # 给调度器看
  data: { ... }                               # 给下游 agent 用
  cost:                                       # 给财务看
    model: claude-sonnet-4-6
    tokens: 3200
    usd: 0.02
  gap_reason: null                            # 给监控台看(status 非 success 时填)
```

### status 枚举(五值)

| 值 | 含义 | 调度器动作 |
|---|---|---|
| `success` | 正常跑完 | data 传下游 |
| `no_data` | 没数据可算(如昨天 0 花销)· 不是错 | 跳过,监控台标 `gap_reason` |
| `external_failure` | 外部依赖挂了(Meta/Google/LLM) | 告警推运营,不推投手 |
| `timeout` | 超时被掐 | 重试 N 次,N 次后转 failed |
| `failed` | 平台自己挂了(bug/DB/...) | 入 dead-letter,告警推 ZJB |

### 四字段的刚需理由

| 字段 | 没有会怎样 |
|---|---|
| status | agent 之间没法协调,监控台一片红,故障被淹没 |
| data | 下游 agent 拿不到数据,孤岛 |
| cost | 财务面板无数据源 |
| gap_reason | 监控台看到空值只能瞎猜,修复慢 |

## 12.3 gap_reason 标准化取值

避免每个 agent 自己编文案,定一组枚举:

```
meta_api_unavailable         Meta 接口没响应
google_api_unavailable       Google 接口没响应
llm_timeout                  LLM 调用超时
site_not_launched            站还没上线
no_spend_yesterday           昨天 0 花销,无数据可算
investor_report_missing      投手/吴玲日报未提交
upstream_failed              上游 agent 失败,我没法跑
category_mismatch            打法包和站品类不匹配
unknown                      兜底,不应该常见
```

新原因要加,走平台侧评审,不允许 agent 代码里散写。

## 12.4 调度方式 · MVP 走甲(进程内)

| 方式 | MVP 选择 | 升级路径 |
|---|---|---|
| 甲 · 进程内调用 | ✅ **MVP** | —— |
| 乙 · HTTP | 后续升级 | 量大、要独立部署时切 |
| 丙 · 队列 | 暂不 | 有异步/削峰需求时 |
| 丁 · gRPC | 不考虑 | —— |

**投手跨机器需求澄清:**
- 投手 A/B/C 办公在哪 → web 浏览器访问平台,和 agent 架构无关
- agent 跑在平台服务器,不在投手电脑
- 投手协作靠共享 data-spine + 权限系统,不靠 agent 跨机器
- 客户站"放客户自己服务器"是 P2 事项,不在 MVP 范围

**MVP 升到乙的改造量:** 只要 §12.1 / §12.2 的字段结构不变,从进程内函数调用换成 HTTP,**业务代码不动**,只改 runtime 调度器。

## 12.5 调度器拿到出参后的动作(伪代码)

```python
result = agent.run(input)

# 1. 先写 data-spine
db.agent_calls.insert(call_id, site_id, agent_name,
                      status=result.status,
                      input_snapshot=input,
                      output_snapshot=result.data,
                      error=result.gap_reason)
if result.cost:
    db.cost_events.insert(site_id, source=agent_name,
                          cost_usd=result.cost.usd, ...)

# 2. 按 status 分支
match result.status:
    case "success":
        trigger_downstream(result.data)          # 传下游
    case "no_data":
        mark_gap_in_dashboard(result.gap_reason) # 监控台标灰
    case "external_failure":
        alert_ops(result.gap_reason)             # 推运营
    case "timeout":
        retry_or_escalate()                      # 重试或升级
    case "failed":
        dead_letter.push(call_id, input)         # 入兜底队列
        alert_owner()                             # 推 ZJB
```

## 12.6 未决(小)

- overrides 字段是否需要白名单(防止投手瞎改把 agent 搞崩)
- cost 里 tokens 对非 LLM agent(如 channel-adapter 拉 Meta 数据)怎么记,是记 API 调用次数还是放空
- 重试次数 N 定多少(初步建议 timeout 3 次 · external_failure 5 次)

# 13. repo 形态与客户剥离边界(05-13 P0-1 收口)

## 13.1 决议:monorepo + 卖站剥离口子

**形态:** monorepo,一个 git repo 装下平台核与所有内部站。
**位置:** `~/Documents/GitHub/multisite-platform/`(与 `~/Documents/GitHub/crave-AI/` 平级,物理隔离硬约束自动满足)。

```
~/Documents/GitHub/
  ├─ crave-AI/                         ← 现有生产资产,只读不动
  └─ multisite-platform/               ← 新平台 monorepo
      ├─ platform-core/                ← 发动机:agents / packages / workflows / data-spine / runtime
      ├─ sites/                        ← 车壳:每站一个目录
      │   ├─ internal-mock-customer/
      │   └─ ...
      └─ tools/                        ← 工程脚手
          └─ customer-export/          ← 客户剥离脚本(P1 工程项,先占位)
```

## 13.2 选 monorepo 不选其他形态的理由

| 判断题 | ZJB 答案 | 推论 |
|---|---|---|
| 投手能不能看别人的站 | 短期不隔离 | 排除"多 repo"(多 repo 是为隔离付的代价) |
| 升级 agent 是否要全站同步 | 是,投手不写 agent | 推荐 monorepo(发动机一改全车升级) |
| 是否会卖站给客户 | 半个月到一个月会卖,卖源码且能拉升级 | 不强制多 repo,但要保留**单站剥离能力** |

monorepo 同时满足"内部统一升级"和"客户剥离单站",是当前条件下唯一不冲突解。

## 13.3 客户剥离边界(硬约束 · 必须从第一天遵守)

卖站给客户 = 把 `sites/{站}/` + 所需 `platform-core/` 子集打包成独立 repo,客户后续通过 sync 脚本(git subtree pull 或同等机制)从主 monorepo 拉 `platform-core/` 升级,客户改的 `site/` 不被覆盖。

**这意味着今天起,代码必须遵守两条边界:**

1. **`platform-core/` 自洽** —— 不能 import / 反向引用 `sites/` 下任何东西。如果需要站特定数据,从 `sites/{site}/config.yaml` 通过 runtime 注入,不在 platform-core 代码里写死 site_id。
2. **`sites/` 调用 platform-core 走稳定接口** —— 只通过 import 平台对外暴露的模块,不许相对路径跳进 platform-core 内部结构乱拿,否则客户拿走后升级会炸。

**P1 工程项(不阻塞当前架构推进):**
- `tools/customer-export/` 脚本:输入站 ID,输出可独立运行的客户 repo 包
- `tools/customer-sync/` 脚本:客户在他自己 repo 里跑,从你这边 pull `platform-core/` 新版本
- platform-core 对外接口稳定性约定(版本号 / 破坏性变更标记)

## 13.4 与 v3 §7 repo 结构的衔接

§7 已经画了 `platform-core-平台内核/` 和 `sites-站群/` 两个一级目录,§13 把它们正式定为 monorepo 内的两个一级目录,**不是两个独立 repo**。§7 注释里"每站一个 repo · 隔离"这句作废,以 §13 为准。

## 13.5 未决(P1+,不阻塞)

- monorepo 工具选型(纯目录 / uv workspace / pnpm workspace),取决于后续技术栈定型
- 客户拿到剥离 repo 后的私有许可证 / 服务条款(法务事项)
- platform-core 对外接口的版本号策略(SemVer 还是 CalVer)

## 13.6 稳定接口第一层已定义(05-13 下午)

§13.3 只立了"platform-core 自洽 + sites 走稳定接口"的硬约束,没定义稳定接口形态。05-13 下午已补:

```
└─────────────────────────────────────────┘
              ↓ 调度
┌─────────────────────────────────────────┐
│ Domain Agents(4 类)                     │ 执行层
│ Traffic Agent(流量·渠道·投放)           │
│ Tech Agent(埋点·速度·API)              │
│ Data Agent(归因·异常·分析)              │
│ Creative Agent(素材·脚本·生产)          │
│ agent 互相影响(虚线):attribution_risk → │
│   降权建议;素材老化信号 → 触发生产        │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ Data Runtime                            │ 数据层
│ 状态写回 · 执行日志 · 事件源 · xlsx/API/logs │
└─────────────────────────────────────────┘
              ↑
        新事件涌现 → 回到事件层
```

## 15.2 员工视角(这一层的本质)

> **员工拿到的是:**
> 任务自动出现 · 状态自动推进 · 历史自动继承 · Agent 自动辅助
> → 员工只负责执行决策

员工不触碰事件层 / Task Engine / Site Runtime Context 任何一层,他们只在"任务推到面前时做决策"这一个点上。所有上游(事件捕获、任务生成、历史继承)和下游(状态写回、日志)都是 Runtime 自动。

## 15.3 能力包三层挂载(关键)

§15 把 §1-§14 定义的三种能力包**挂在不同深度**,不是同层:

| 能力包 | 挂载位置 | 作用 |
|---|---|---|
| **打法包** | Site Runtime Context(上下文层) | 告诉 agent「这个站怎么决策」——阈值、预测程度、用哪些 agent |
| **产出包** | Runtime Task Engine(引擎层) | 作为可调度的任务类型存在——事件触发后任务引擎决定调哪个子包 |
| **内容包** | Agent 执行层入参 | Creative Agent 跑的时候读内容包,决定话术调性 |

**一句话:** 打法包是上下文,产出包是任务,内容包是素材。三层对应 Runtime OS 的三个不同深度。

**与 §4 的关系:** §4 只定义了"双层能力包"(打法包 + 产出包),§15 追认为**三层**(打法包 + 产出包 + 内容包),并把三层的**运行时挂载点**定死。

## 15.4 事件 → agent 的跨域互相影响(虚线)

图里的两条虚线是例子,不是穷举:

- `Data Agent.attribution_risk` → Traffic Agent 降权建议
- `Creative Agent.素材老化信号` → 触发自己的生产流水线

**Why 虚线:** agent 互相影响是 Runtime 的常态,但不走入参出参的硬合约(§12),走事件层广播。任一 agent 的输出都可能被其他 agent 订阅为事件源。

## 15.5 §15 与 §12 I/O 合约的关系

§12 定义的是**单个 agent 调用**的入参/出参 schema(site_id + time_window + upstream_output + overrides / status + data + cost + gap_reason)。

§15 定义的是**谁在什么时候调这个 agent**:
- 由事件层捕获触发
- 由 Task Engine 调度(决定优先级、重试、回滚)
- Agent 执行时读 Site Runtime Context 注入参数

§12 是"怎么调",§15 是"什么时候调 / 谁调 / 调完写哪"。
